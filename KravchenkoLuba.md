# Работа с Git
## 1.Проверка наличия установленного Git
В терминале нужно выполнить команду git version.
Если git установлен, появится сообщение с информацией о версии.Иначе будет сообщение об ошибке.
## 2.Установка Git
Загружаем последнюю версию git с сайта git-scm.com\downloads
Устанавливаем с настройками по умолчанию
## 3.Настройка Git
При первом использовании git неоходимо представиться. Для этого нужно ввести в терминале две команды
```
git config --global user.name «Ваше имя»
git config --global user.email ваша "почта@example.com"
```

## 4.Инициализация репозитория
Получить репозиторий можно двумя способами.

### Первый способ:
* создать новую папку (можно на рабочем столе, в моих документах и т.д. она будет репозиторием)
* открыть ее в VSCode 
* перейти в терминал (View-Terminal)
* ввести в терминале команду git init
### Второй способ:
Клонировать уже существующий репозиторий.

__Git init__ - Инициализация: указываем папку, в которой
git начнёт отслеживать изменения.
В папке создаётся скрытая папка .git

## 5.Создание файла
Во вкладке File выбираем New File, вводим его название и нажимаем Greate New. Также можно использовать быструю клавишу New File, которая находится рядом с названием папки.
Чтобы использовать разметку Markdown, делаем расширение файла md. 
## 6.Git add
Для того чтобы git начал отслеживать наш файл, необходимо добавить его с помощью команды Git add.
ВВодим в терминал команду Git add + название файла (например:git add Git-education.md)

__Git add__ добавляет содержимое рабочего каталога 
в индекс (staging area) для последующего коммита. Эта команда дается после добавления
файлов. Писать название целиком не обязательно: терминал дозаполнит данные автоматически.
## 7.Git commit -m 

Git commit -m "Текст коммита"

Затем нам необходимо зафиксировать то, что мы создали файл,делаем это с помощью команды Git commit.
Вводим в терминал команду Git commit -m "Добавили файл".

__Git commit__ - зафиксировать или сохранить.
По умолчанию git commit использует лишь этот индекс, так что вы можете использовать git add 
для сборки слепка вашего следующего коммита.
Команда git commit берёт все данные, добавленные в индекс с помощью git add, и сохраняет их
слепок во внутренней базе данных, а затем сдвигает указатель текущей ветки на этот слепок.
## 8.Git commit -am

Git commit -am "Текст коммита" или Git commit -a -m "Текст коммита"

Можно объединить две команды git add и git commit в одну git commit -am.
В этом случае сохраняться будут все изменения всех файлов.
## 9.Автосохранение или CTRL+S
Начинаем работу непосредственно в файле, вводим в первую строку заголовок "Работа с Git". Чтобы можно было создать коммит на это действие нужно сохранить изменения файла с помощью клавиш CTRL+S.

Также можно включить автосохранение во вкладкe File выбрать пункт Auto Save.
## 10.Git status
После сохранения изменений файла введем в терминале команду git status и появится информация о том, что есть изменения, которые нужно закоммитить. 
Введем в терминал команду git commit -am "Работа с Git".
Затем снова введем git status, появится информация о том, что у нас нет незакоммиченых изменений.

__Git status__ - Показывает текущее состояние гита, есть 
ли изменения, которые нужно закоммитить
(сохранить).
## 11.Git diff
Далее добавим в наш файл пункт 1 и введем в терминале команду git diff, в терминале появится информация о добавленых строках.

__Git diff__ показывает разницу между текущим файлом
и сохранённым.
## 12.Git log
Эта команда вызывает журнал изменений.
Перед переключением версии файла в Git
используйте команду git log, чтобы увидеть
количество сохранений.
Добавим в наш файл еще несколько пунктов, после каждого создавая коммит. Затем введем команду  Git log и сможем увидеть журнал изменений.
## 13.Git checkout
Переключение между версиями.
Для работы нужно указать не только
интересующий вас коммит, но и вернуться 
в тот, где работаем, при помощи команды 
git checkout master.
## 14.Подсказки
Git отслеживает файлы по имени! 
Если изменить имя файла, необходимо добавить файл с новый именем + git commit.

Чтобы вызвать ранее введённую команду,
пользуемся стрелками на клавиатуре.
Перебираем недавно введённые команды
нажатием стрелки «вверх».

## 16.Git branch

**Git branch** - команда, которая выводит список всех имеющихся веток. Ветка, на которой мы находимся в данный момент отмечена *.

**Git branch имя новой ветки** - команда для создания новой ветки. Делать это надо в папке с репозиторием.

**Git branch -d** - команда для удаления ветки. Как правило, ветка удаляется после слияния, так как больше не нужна.

Название ветки должно отражать ее предназначение.

Несколько веток необходимо для ведения черновиков и когда несколько человек работают над разными частями одного проэкта.
## 17.Git checkout
С помощью Git checkout можно переходить в разные состояния файла по коммитам, набрав команду **Git checkout первые 4 символа из номера коммита**. Предварительно нужно воспользоваться командой Git log. Для возвращения в актуальное состояние пользуемся командой **Git checkout master**.

С помощью Git checkout также можно переходить на разные ветки,набрав команду **Git checkout имя ветки**. Данная команда не сработает, если на текущей ветке файл изменен (справа от названия файла стоит буква М), но не закоммичен. 


## 18.Git log
Команда **git log** покажет состояние более новых
версий проекта. Но если вызвать эту коgitманду из
самой «свежей» ветки, мы не увидим исходного
файла.
**Git log --graph** -  команда, которая позволяет отобразить коммиты в виде дерева.

Если после команд Git log\Git log graph не вводится текст в теминале, нужно нажать клавишу Q (при этом должна быть включена английская раскладка)

## 19.Git merge
Допустим, черновики нас полностью устраивают и нам нужно внести изменения, чтобы
информация появилась в чистовике. Для этого есть команда git merge. **Git merge имя ветки** - команда, которая позволяет слить любую ветку с текущей. Вызываем эту команду находясь в главной ветке, куда хотим залить второстепенную ветку.

При работе в двух ветках одновременно может
возникнуть ситуация, когда в одной и другой
ветке мы по-разному изменили блок текста.
Если затем мы попробуем слить эти ветки, Git
сообщит о конфликте и предложит выбрать,
какие же изменения записать:
* изменения текущей ветки (главной, на которой мы сейчас находимся)
* изменения вливаемой ветки (второстепенной)
* изменения обеих веток
Поэтому у проекта в репозитории должен быть один
ответственный пользователь, наделённый правом проводить
слияния и разрешать конфликты.

## 20.Работа с удаленным репозиторием.
Для работы с удаленным репозиторием создаем аккаунт на Git Hub. Затем создаем новый репозиторий на Git Hub и локальный репозиторий на своем компьютере, "знакомим" их с помощью подсказок Git Hub и работаем. Нам пригодятся следующие команды:

__Git clone__ - команда, которая копирует внешний репозиторий на свой ПК. Она не только
загружает все изменения, но и пытается слить 
все ветки на локальном компьютере и в
удаленном репозитории. После данной команды пишется адрес репозитория, который хотим клонировать, обычно он копируется из Git Hub.

Отправить свою версию репозитория во
внешний репозиторий поможет команда __Git
push__. При первом её использовании нужна авторизация.

**Git pull** - команда, которая позволяет скачать все 
из текущего репозитория и автоматически
сделать merge с нашей версией.

Если нам нужно поработать в чьем-то репозитории, заходим на страницу нужного репозитория и нажимаем кнопку Fork. C ее помощью мы копируем нужный репозиторий к себе на Git Hub, затем клонируем его, вносим нужные изменения, отправляем их на Git Hub (push) и делаем запрос обладателю этого репозитория на внесение наших изменений.

**Pull request** - команда для предложения изменений, запрос на вливание изменений в репозиторий.
В больших компаниях один ответственный за проект создает аккаунт. Другие пользователи дают
команду pull request. Сначала
пользователь копирует репозиторий на свой компьютер, делает fork репозитория, затем
клонирует версию на своём ПК, создаёт ветку с предлагаемыми изменениями, отправляет
изменения командой push в свой аккаунт на GitHub и даёт команду pull request.
