## Базовые комманды Git
*сочетание ctrl+S сохраняет изменения файла в VS Code*
* **git config --global |args** - команда для настройки Git(редактирует файл настроек /.config/git/config), используется в терминале Git Bash, имеет следующие аргументы: 
    *  **user.name "Type your name"** - задает имя пользователя, которое будет указываться в коммитах и редактированиях
    * **user email example@mail.com** - аналогично, задает почтовый адрес пользователя в подписи
    * **core.editor "'full_path_to_editor_executable_file' -w"** - изменяет рабочий текстовый редактор на указанный, вместо того, который стоит по умолчанию.
* **git init** -команда, инициализирующая локальный репозиторий в текущей папке(по сути, создает скрытый файл .git c технической информацией)
* **git add [path-filename]** - команда, добавляющая файл или папку к отслеживанию
  *используем Tab при наборе для выбора пути*
* **git add --update [path_spec]** - команда, обновляющая индекс, чтобы привести его в соответствие к рабочему дереву в указанной папке. Если путь к папке не задан, обновляется индекс по всем файлам рабочего дерева.
* **git commit -m""** -сохранение текущего состояния отслеживаемых файлов, m - это комментарий . Так же возможен аргумент **-a** (сокращено от **--all**), который автоматически коммитит все измененные или удаленные файлы из индекса.
* **git commit reuse-message=[commit hash number]** - коммит с таким аргументом позволяет сделать коммит с той же пометкой,временем,и подписью юзера, что была в коммите с указанным хэшом. Полезно, если забыли добавить какой-то файл к коммиту, и нужен рекоммит без лишних записей в логе. Вариация **reedit-message=** позволит также отредактировать пометку.
* **git diff** - cравнение между коммитами или файлами
* **git checkout |commit number|** - возврат состояния файлов к определенному коммиту.
* **git checkout master** - возврат к ветке мастера

## Работа с ветками
### вспомогательные команды
* **git branch** - вывести список существующих веток. Звездочкой помечена текущая.
* **git branch -d branch_name** - удалить ветку с указанным именем.
***Внимание!** Перед удалением ветки, убедитесь, что вы сохранили нужные данный(внешним бэкапом или слиянием веток)*
* **git log --graph** - вариант отображения лога, который графически показывает все форки и слияния веток.
* *также можно создать файл __".gitignore"__ в рабочей папке* - если в вашей папке есть какие-то файлы, которые гит отслеживать не должен, в этом файле нужно перечислить названия игнорируемых файлов.
* **git checkout master** - вернуться в главную рабочую ветку(также может иметь другие названия по умолчанию, например, __main__).

### Создание новой ветки
Чтобы создать новую ветку,используем следующую команду  ("branch_name" - любое название):

    git branch branch_name
В нее скопируется содержимое текущей ветки, с учетом последнего коммита.    
### Переключение между ветками
Чтобы переключиться на другую ветку мы используем следующую команду ("branch_name" - имя ветки, на которую переходим):

    git checkout branch_name
### Слияние веток
Чтобы слить две ветки вместе, мы используем следующую команду:
    
    git merge branch_name
**Важные моменты:**
* изменения сливаемой ветки будут добавлены в текущую.
* по-хорошему, вливаемые изменения не должны задевать изменения тех же строк, что уже менялись в текущей ветки.
* "branch_name" - имя ветки, откуда сливаются изменения.

### Конфликты при слиянии
При возникновении конфликтов между сливаемыми ветками, VS Code может предложить следующие варианты ручного разрешения конфликтов:
* оставить изменения из текущей ветки(например, master) 
* перезаписать информацию на изменения импортируемой ветки (import change)
* оставить обе версии в файле, для разрешения конфликта в будущем(both modified) .С этим можно коммитить.
* compare - сравнить, открывает оба файла для просмотра, с цветовой кодировкой изменений (красные-отсутствующие, зеленые-новые). Файл в VS Code будет сохранен с двумя версиями, как в 3-м варианте. 

### Методы слияния
При сложном ветвлении, Git может применять быстрые стратегии слияния. Например, если изменения уже есть в мастере, конфликты будут автоматически разрешаться в их  при слиянии в побочных ветках.
Слияние по  'ort' стратегии пропускает промежуточные ветви и конфликты с ними, если сливается мастер с крайней веткой.

### Работа с удаленными репозиториями.
1. Копирование репозитория по ссылке __(git clone)__:
чужой удаленный репозиторий->свой локальный репозиторий

2. Создание удаленного репозитория(git pull, git push)
  2.1  Создание аккаунта на GitHub
  2.2  Создание репозитория на github
  2.3  Создание и подключение локального репозитория

    Создание нового локального через коммандую строку 


        echo "# crashtest-course" >> README.md
        git init
        git add README.md
        git commit -m "first commit"
        git branch -M main
        git remote add origin https://github.comEvercrow/ crashtest-course.git
        git push -u origin main

 или уже сущевствующий 

     git remote add origin https://github.com/Evercrow/crashtest-course.git
     git branch -M main
     git push -u origin main  
 аргумент -u указывает на первоначальный push и pull
 * __git pull__ - выгрузка всех данных с подключенного по remote add репозитория  
 * __git push__ - загрузка изменений с локального репозитория на подключенный удаленный , нужна авторизации
 * __git clone origin (github url)__ - копирование данных с репозитория по ссылке, автоматически подпапку в рабочей папке с названием репозитория.

Для подключения к чужому проекту на github ,  нужно зайти на удаленный репозиторий и сделать Fork . Это создаст копию удаленного репозитория в моем профиле github, с которым можно работать в локальном репозитории и заливать изменения. Github автоматически отслеживает изменения и позволяет сделать pull request хозяину репозитория для вливания ваших изменений.
