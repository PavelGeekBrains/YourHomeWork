### ТУТОРИАЛ РАБОТЫ В GIT. *IntoToVersControl*
# *Конспект первого семинара* 
## Общая информация
---
* **Git** - система контроля и управления версиями файлов
* **Visual Studio Code** - редактор исходного кода, позволяяющий работать с файлами и терминалом, что на данном этапе обучения нам и нужно
* **.md** - формат текстовых файлов, позволяющий внутри файла форматировать текст, используя синтаксис *Markdown*
* **Репозиторий** - пространство, в котором работает, которое отслеживает система контроля версий  
## Изученные команды системы контроля версий Git
---
* **git config --global user.name** *«YourName»* - команда, позволяющая представиться git'у
* **git config --global user.email** *YourMail* - команда, позволяющая указать git'у адрес своей электронной почты
* **git init** - команда, позволяющая создать репозиторий в текущей папке
* **git version** - команда, позволяющая проверить версию git'а
* **git add .\fileName** - позволяет добавить файл в отслеживаемые git'ом
* **git commit -m "your note"** - позволяет зафиксировать изменения файла и добавить заметку для сохраненной версии
* **git log** - позволяет вызвать журнал сохраненных версий (коммитов) файла *(желтым цветом в журнале выделены коды коммитов)*
* **git status** - позволяет отобразить текущее состояние репозитория
* **git checkout** *versCode* - позволяет вернуть состояние файла, соответствующее указанной версии
* **git checkout master** - позволяет вернуть актуальное состояние файла
* **git diff** - позволяет посмотреть разницу между текущим состоянием файла и его зафиксированной версией
## Алгоритм работы с Git, закрепленный на семинаре
---
*Исходим из того, что Git и Visual Studio Code уже установлены*  
  
1. Необходимо **создать директорию**, в которой планируется отслеживать версии файла *(имя папки нужно задать латиницей без использования пробелов)*
2. Запустить **Visial Studio Code**
3. Открыть **проводник** *(иконка проводника находится на левой боковой панели VSC)*
4. Выбрать, созданную для отслеживания, **папку**
5. Создать **файл .md**-разрешения *(имя задать латиницей без пробелов)*
6. Вызвать **терминал** *(меню "терминал", расположенное на верхней панели VSC)*
7. С помощью терминала и соответствующих команд, **представиться** и **указать свой e-mail** git'у
8. **Создать репозиторий** в папке с помощью соответствующей команды
9. **Начать отслеживать созданный файл** с помощью соответствующей команды
10. **Зафиксировать версию файла** с комментарием об успешном начале работы *:)* с помощью соответствующей команды
11. Вызвать **журнал сохраненных версий**, чтобы проверить успешно ли сохранилась первая версия файла *(выйти из журнала можно с помощью клавиши "Q")*
12. **Внести** желаемые **изменения** в файл 
13. **Сохранить изменения**, используя *Ctrl+S*
14. Cнова **добавить файл в отслеживаемые**
15. **Зафиксировать текущую версию** файла

*Пункты 12-15 необходимо повторить столько раз, сколько значимых изменений мы вносим в файл*

## Другая полезная информация, приобретенная на семинаре
---
* Закрыть папку можно с помощью одного из пунктов меню "Файл"
* Помимо команд указанных в алгоритме, полезно использовать такие команды как **git diff**, **git status** для получения информации о наличии изменений в репозитории и в файле
* Команда **cd..** позволяет перейти в папку, находящуюся выше в иерархии
* Второй раз применять **init** в папке не нужно, даже после выхода и работы в другой директории 
* Удаление папки **.git** из репозитория позволит отменить и "забыть" всю проделанную в git работу, касающуюся этого репозитория
* Маркер "**U**" рядом с именем файла говорит о том, что файл не отслеживается
* Маркер в виде **белой точки**, говорит о том, что файл изменён, но не сохранён
* Маркер "**М**" говорит о том, что файл изменён, сохранён, но версия не зафиксирован
* Маркер "**А**" говорит о том, что файл добавлен в отслеживаемые 
* Команда **clear** очищает терминал *(не является командой git'а)*
* **Стрелки вверх** и **вниз** помогают переключаться между командами, введенными 
ранее
* В некоторых случаях **Tab** удобно использовать как автоподбор
* С помощью команды **git checkout** можно перемещаться между коммитами, введя через пробел от команды код нужного коммита *(для возвращения к последнему коммиту на месте кода нужно ввести **master**)*
* После перехода на более раннюю версию файла с помощью **git checkout**, что-либо менять в файле, не вернувшись на актуальную версию, **крайне нежелательно**
* **git commit --amend -m** "*corrected note*" - команда, которая позволяет исправить комментарий к текущему коммиту
* При **переименовывании файла**, git воспринимает это как удаление из репозитория файла со старым названием и создание в репозитории файла с новым названием, при этом статус файла меняется на **"Untracked"** *(при переходе на более ранний коммит, относительно того в котором файл был переименован, имя файла сменится на старое)*
* Терминал, действительно, иногда вредничает и не даёт выйти из журнала log, но это спокойно лечится перезапуском VSC

### ТУТОРИАЛ РАБОТЫ В GIT. *IntoToVersControl*
# *Конспект второго семинара* 
## Изученные команды 
---
* **git branch** - команда, которая позволяет посмотреть список веток  
* **git branch  branch_name** - команда, позволяющая создать ветку с именем "branch_name" 
* **git merge branch_name**  - команда, осуществляющая слияние текущей ветки с веткой "branch_name"  
* **git branch -d branch_name** - позволяет удалить ветку "branch_name"
* **git log --graph** - команда, графически показывающая иерархию веток и коммиты  
## Алгоритм работы с ветками
---
1. Фиксируем состояние, после которого хотели бы начать работу с ветками
2. Создаём новую ветку с помощью команды **git branch branch_name**
3. Переходим на новую ветку с помощью команды **git checkout branch_name**
4. Вносим желаемые изменения
5. Сохраняем изменения с помощью *Ctrl+S*
6. Добавляем файл в отслеживаемые с помощью **git add**
7. Фиксируем с помощью **git commit** 
8. Переходим на ветку *master*
9. ##### *При необходимости вносим изменение в файл на ветке мастер*
10. ##### *Добавляем в отслеживаемые*
11. ##### *Фиксируем*
12. Проводим слияние ветки *master* с нужной нам веткой, с помощью команды **git merge branch_name**
13. Вслучае, если слияние прошло с конфликтом, разрешаем конфликт и делаем соответствующий коммит  

##### *Пункты 4 - 7 и 9 - 11 необходимо повторить столько раз, сколько изменений в соответствующие ветки мы хотим внести и зафиксировать*

*В случае fast-forward слияния, нет необходимости в дополнительном коммите. В противном случае, но при условии, что слияние произошло без конфликта, происходит автокоммит, и необходимость в дополнительном коммите также отпадает*
## Другая полезная информация со второго семинара
---
* При *fast-forward* слиянии, ветка master просто "догоняет" ту ветку в которой мы работали
* Если мы фиксировали изменения и в побочной ветке, а затем и в ветке master, при слиянии может возникнуть конфликт, который необходимо разрешить и сделать коммит с заметкой о разрешении конфликта
* Если в аналогичной ситуации конфликта не возникло, случится автокоммит, что подстегнёт весело и задорно работать дальше
* Не нужно воспринимать конфликт как ошибку
* При возникновении конфликта будет предложено три варианта его разрешения
* При выборе варианта *Current Change* мы примем изменения, которые были сделаны только в текущей ветке
* При выборе варианта "Incoming Change" мы примем изменения, которые были сделаны в присоединяющейся ветке
* Вариант *Accept Both Changes* считается каноном и позволяет сохранить изменения обоих веток

# *Конспект третьего семинара* 
## Изученные команды 
---
* **git branch -M master** - благодаря этой команде, ветка master будет основной и в локальном репозитории и в удалённом
* **git remote add origin ссылка** - с помощью этой команды, даём понять, что хотим подвязать наш репозиторий с удалённым
* **git push -u origin master** - "толкаем"(push) на github ветку ветку мастер, обозначая её как основную (origin) и делая её доступной к отправке (-u (upstream))
* **git checkout -b имя_ветки** - создаём ветку и сразу же на неё переключаемся
* **git push --set-upstream origin имя_ветки** - делаем указанную ветку доступной для отправки  

## Алгоритм создания удаленного репозитория
---
### **Вариант 1. Создаём репозиторий через командную строку**
1. Создаем в нужной папке README.md, прописываем там всё, что нужно
2. Инициализируем репозиторий
3. Добавляем в отслеживаемые README.md
4. Делаем первый коммит
5. Вводим команду **git branch -M**, но вместо main прописываем **master** *(таким образом главной и в локальном, и в удаленном репозиториях будет ветка **master**)*
6. Вводим **git remote add origin *ссылка***, то есть говорим, что хотим подвязать наш репозиторий с удаленным, который находится по ссылке
7. Вводим **git push -u origin**, но вместо main снова прописываем **master** (**push** - “толкаем” на сервер; **-u** говорит о том, что указанная ветка доступна к отправке (*upstream*); **origin** говорит о том, что базовая ветка здесь, будет базовой и в удаленном репозитории)  

### **Вариант 2. Делаем любой репозиторий, в котором уже работали, удаленным**
 * Работая в репозитории, который хотим сделать удаленным, с помощью терминала вводим комадны из пунктов 6, 5, 7 предыдущего алгоритма.

 ## Алгоритм форка чужого репозитория
 1. На GitHub находим нужный нам репозиторий у кого-либо
 2. Усердно ищем и находим кнопку **Fork**
 3. При желании или необходимости переименовываем репозиторий
 4. Жмём на **Create Fork**
 5. Быстро и удобно создаем локальную версию форкнутого репозитория c помощью кнопки "Clone Git repository" в VSC
 6. Выбираем нужный репозиторий из списка имеющихся на аккаунте
 7. Выбираем папку, в которой хотели бы его разместить
 8. Перед тем как делать изменения нужно перейти в новую ветку
 9. Работаем в новой ветке, делаем коммиты
 10. Разрешаем апстримить нашу доп ветку и отправляем её к нам на github 
 11. В нашем удаленном репозитории появилась новая ветка и стала доступна кнопка **Compare & Pull request**, которую и нажимаем
 12. При необходимости меняем имя и прикрепляем сообщение
 13. Жмём **Create pull request**
 14. На следующей странице есть возможность обмениваться сообщениями с автором репозитория, пока наши изменения не будут приняты (или отвергнуты)

 ## Другая полезная информация с третьего семинара
 ---
 * У одного локального репозитория может быть связь только с одним удаленным
 * Чтобы внести изменения в файл из репозитория на github, специально придумана и внедрена кнопка под кодовым названием Карандашик, жмём (edit this file)
 * Изменения на сделанные в файле репозитория на github необходимо сразу коммитить с помощью окна Commit Changes, находящегося сразу под окном редактора файла
 * *Первое поле ввода, в котором сейчас написано “Update README.md”, это сообщение, которое мы вводили в терминале при коммите (-m “…”)*
 * Кнопка **Commit changes** проводит команды **git add** и **git commit**, при этом локальный репозиторий пока не знает об изменениях, сделанных в удаленном репозитории
 * Чтобы актуальная версия попала в наш локальный репозиторий, в терминале используем команду **git pull**
 * Кнопка **Fork** позволяет создать копию чьего-то репозитория у себя на github’е, изменять у себя (в доп ветке) и затем запрашивать право на слияние с репозиторием автора
 * Находясь у себя на аккаунте на странице форкнутого репозитория, в левой верхней части экрана указано с кого форкнут репозиторий
 * Когда создаем репозиторий, первоисточником является локальный репозиторий. Когда клонируем репозиторий первоисточником является удаленный репозиторий
 * Перед тем как делать изменения, работая в форкнутом репозитории, нужно перейти в новую ветку
 * Работая в новой ветке в форкнутом репозитории, не получится отправить изменения просто командой git push, так как необходимо заапстримить ветку (сделать доступной для отправки) **git push —set-upstream origin имя_ветки**
